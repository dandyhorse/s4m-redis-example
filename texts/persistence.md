## 14 — Persistence: дополнения

### AOF fsync-политики

Ключевая конфигурация, определяющая реальный RPO:

- `appendfsync always` — fsync после каждой записи. Максимальная надёжность, но убивает throughput.
- `appendfsync everysec` — fsync раз в секунду. **Дефолт и рекомендация.** Потеря — максимум ~1 сек.
- `appendfsync no` — отдаём на откуп ОС (обычно ~30 сек). Быстро, но RPO непредсказуем.

### RDB + AOF вместе

Redis поддерживает и сам рекомендует использовать оба механизма одновременно. При старте восстановление идёт из AOF (он полнее), а RDB используется как «быстрый бэкап» для катастрофических сценариев или для реплик.

### Мульти-файловый AOF (Redis 7.0+)

Вместо одного гигантского файла теперь `appendonlydir/` с base-файлом (в RDB-формате!) + инкрементальными AOF-файлами. Это ускоряет rewrite и восстановление. Старые статьи на Хабре описывают однофайловую модель — она уже неактуальна.

### Fork / COW — практический нюанс

В проде стоит следить за `used_memory` vs доступной RAM. Worst case при COW — если во время BGSAVE/BGREWRITEAOF переписываются *все* страницы, потребление памяти может вырасти почти x2. Поэтому общая рекомендация — держать `maxmemory` не более 45–50% от доступной RAM на машине.

### Конфигурация RDB-триггеров

```
save 3600 1      # снапшот если >= 1 изменение за 3600 сек
save 300 100     # или >= 100 изменений за 300 сек
save 60 10000    # или >= 10000 изменений за 60 сек
```

Несколько `save`-директив работают по логике «ИЛИ» — снапшот создаётся при срабатывании любого из условий.
