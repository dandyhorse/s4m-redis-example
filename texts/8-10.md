## 08 — Geospatial

- **Geospatial — это надстройка над Sorted Set.** Координаты кодируются через geohash в score, поэтому удаление члена делается через `ZREM`, а не через отдельную geo-команду.
- **Точность — 52-битная сетка.** Из-за квантования и округления при преобразовании туда-обратно теряется ~7.5 см — для большинства задач это несущественно.
- **`GEOSEARCH` заменяет `GEORADIUSBYMEMBER` и `GEORADIUS`.** Принимает как координаты, так и имя члена set'а в качестве центра поиска, и поддерживает поиск по радиусу или прямоугольнику.
- **Geohash — строка из 11 символов.** Можно обрезать справа (теряя точность, но сохраняя район). Похожий префикс = рядом, но рядом ≠ похожий префикс.

## 09 — HyperLogLog (HLL)

- **HLL считает приблизительное количество уникальных элементов.** Это вероятностная структура данных — она не хранит сами элементы, только «отпечаток» множества.
- **Три команды — и всё:** `PFADD` (добавить), `PFCOUNT` (посчитать уникальные), `PFMERGE` (объединить несколько HLL).
- **Занимает максимум ~12 КБ** независимо от количества добавленных элементов. Стандартная погрешность — 0.81%.
- **Типичный use-case** — подсчёт уникальных посетителей, уникальных IP, уникальных запросов, где точное число не критично, а экономия памяти — критична.

## 10 — Streams

- **Streams — это append-only лог** с автогенерируемыми ID вида `<timestamp>-<seq>`. Каждое сообщение — набор полей (field-value пар), как в хеше.
- **`XADD` с `*`** — Redis сам назначает ID (timestamp + порядковый номер). Гарантируется уникальность и порядок.
- **`XRANGE` / `XREVRANGE`** — чтение диапазона сообщений. `-` и `+` означают «с начала» и «до конца» соответственно.
- **Streams мощнее Pub/Sub**: сообщения персистентны, поддерживают consumer groups (группы потребителей), acknowledgment, и чтение с произвольной позиции — то есть потребитель может подключиться позже и прочитать историю.

### Streams vs Pub/Sub — чем платим за «мощь»

| | Pub/Sub | Streams |
|---|---|---|
| **Память** | Не хранит ничего — fire & forget | Сообщения копятся на диске/в памяти, нужно следить за `MAXLEN` / `MINID` или тримить вручную |
| **Латентность** | Минимальная — сообщение летит подписчику напрямую | Чуть выше — потребитель делает `XREAD`/`XREADGROUP`, это pull-модель (или блокирующий poll) |
| **Простота** | `SUBSCRIBE` + `PUBLISH` — две команды | Consumer groups, PEL (pending entries list), `XACK`, `XCLAIM`, `XAUTOCLAIM` — ощутимо больше концепций |
| **CPU / нагрузка** | Одна операция на публикацию | `XADD` пишет в radix tree + обновляет метаданные группы; тримминг — отдельная работа |
| **Масштабирование fan-out** | Нативный broadcast — каждый подписчик получает копию бесплатно | Для broadcast нужно либо несколько групп, либо каждый читает `XRANGE` сам — больше операций |

### Подробнее о минусах Streams относительно Pub/Sub

**1. Память.** Pub/Sub — fire & forget, после доставки сообщение исчезает. Streams всё копит. Если не ставить `MAXLEN` при `XADD` или не тримить через `XTRIM`, стрим будет расти бесконечно.

**2. Латентность.** Pub/Sub — чистый push, сообщение летит подписчику моментально. Streams — pull-модель: потребитель сам вызывает `XREAD` (можно блокирующий, но это всё равно poll, а не push). На практике разница в микросекундах, но для real-time чатов или тикеров Pub/Sub честнее.

**3. Сложность.** Pub/Sub — две команды (`SUBSCRIBE`, `PUBLISH`). У Streams — consumer groups, PEL (pending entries list), `XACK`, `XCLAIM`, `XAUTOCLAIM`, тримминг. Порог входа ощутимо выше, больше мест где можно ошибиться (забыл `XACK` — PEL растёт, забыл тримить — память растёт).

**4. Fan-out.** В Pub/Sub каждый подписчик получает копию сообщения «бесплатно» — нативный broadcast. В Streams для этого нужно либо создавать отдельную consumer group на каждого «подписчика», либо каждый читает `XRANGE` самостоятельно — это больше операций на сервере.

**Вывод:** Streams не убивает Pub/Sub, они для разных задач. Pub/Sub — когда потеря сообщения допустима и важна скорость/простота (уведомления, инвалидация кеша). Streams — когда нужна надёжность, история и гарантия обработки (очереди задач, event sourcing).
